Project: High Speed Document RAG Pipeline (LangChain & Groq)
Architecture Overview – Here I have developed a Retrieval-Augmented Generation (RAG) system utilizing LangChain and Groq to enable low latency conversational Q&A on unstructured PDF and TXT data. It is optimized to give instant result by first chunking, vectorizing and then indexing the content. This allows the AI to find answers in milliseconds without rereading the entire file every time a question is asked. I had to use the fitz (PyMuPDF) library because standard PDF loaders were failing to capture the formatting of multi-column research papers.
Technical Implementation Details * Retrieval Logic: I used FAISS (Facebook AI Similarity Search) for vector storage. Text ingestion uses a RecursiveCharacterTextSplitter with a specific 1000-character chunk size and 100-character overlap to maintain semantic context (which is how the latter chunks use the former chunk to keep the context) across boundaries.
•	Embedding Strategy: Utilized the intfloat/e5-large-v2 model via HuggingFaceEmbeddings. I chose this model over other standard models because of its higher dimensional accuracy during similarity search. Similarity Search is basically the similar chunks based on question we look into the vector database where contexts of PDF is stored in form of vector.
•	Inference Engine: Integrated the Llama-3.3-70b-versatile model via the Groq API. This setup uses LPUs (Language Processing Units) to get answers instantly, making it way faster than standard cloud services.
•	Memory Management: I used ConversationSummaryBufferMemory. Unlike standard buffer memory, this summarizes older interactions while keeping recent exchanges in raw format and also maintaining a strict 2000-token limit to optimize the context window, this keeps the conversation more relevant. I have also set the temperature to 0.0 in the ChatGroq config to stop the model from hallucinating details not present in the uploaded PDFs.
•	Built-in 'File Memory' System: I implemented a digital fingerprinting system (MD5 hashing) for uploads. By creating a unique ID for every file uploaded, the app can instantly recognize if that document has been uploaded before. This lets it pull the data from its memory instead of wasting time in rereading the entire file.
•	Fixed Cloud Deployment Issues: Resolved a major environment bug on Streamlit Cloud by manually patching the backend library (Torch). I wrote a script which pre-loads the missing modules, ensuring the AI model starts up smoothly every time without crashing.
•	State & Patching: The torch.classes patch was necessary because Streamlit Cloud was throwing an attribute error during the HuggingFace model initialization. I fixed streamlit cloud deployment bugs and also added a 'file recognition' system which helps the app remember the documents uploaded to it. This prevents the system from wasting time by re-processing the same file twice.
The Functional Result The system handles complex, multi-turn dialogues by rephrasing follow-up questions into standalone queries through a ConversationalRetrievalChain, ensuring that the LLM remains grounded strictly in the provided document context.
